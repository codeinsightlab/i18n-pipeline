## 一、当前定位

`i18n-pipeline` 是一个基于 Node.js + TypeScript 的 CLI 工具，用于帮助旧项目做国际化改造前的：

- 中文扫描
- 文本提取
- 资源生成
- 保守替换
- 试跑报告输出

它当前不是 AST 工具，不是翻译服务，不是 UI 管理平台。

---

## 二、必须遵守的总原则

1. 不引入 AST
2. 默认行为必须简单稳定
3. key 稳定性优先
4. append-only，不自动删除旧 key
5. 不做语义 key 自动生成
6. 不做翻译服务
7. 不过度设计
8. 优先真实项目收益
9. 宁可漏，不可错替换
10. 不要轻易扩大替换场景

## 三、当前已完成能力

### 1. CLI 命令

当前支持：

- `i18n scan`
- `i18n extract`
- `i18n replace`
- `i18n run`
- `i18n apply`

说明：

- `scan`：扫描候选
- `extract`：提取并写资源
- `replace`：替换代码，可配合 `--dry-run`
- `run`：`scan -> extract(内存模式) -> replace --dry-run`
- `apply`：`extract -> replace`

### 2. 通用参数

当前支持：

- `--dir <path>`
- `--output <file>`
- `--report <file>`
- `--dry-run`
- `--debug`
- `--help`
- `--version`
- `--structure single|module-dir`
- `--mode merge|clean`
- `--git-check warn|strict|off`

注意：

- 当前主流程默认是 append-only，默认模式按 `merge` 理解
- `run` 为纯预演，不写资源文件
- `apply` 会真正写资源并改代码

### 3. 资源结构

支持两种结构：

- `single`
  - 默认输出：`./i18n/zh.json`
- `module-dir`
  - 例如：
    - `i18n/system/user/zh.json`
    - `i18n/plugins/download/zh.json`

内部统一使用：

- `Map<string, string>` // key -> text

### 4. key 规则

当前 key 规则为：

- `<modulePrefix>.auto_<NNN>`

例如：

- `module.auto_001`
- `system.user.auto_001`
- `plugins.download.auto_002`

规则要求：

1. 优先复用已有 key
2. 新文本才新增 key
3. append-only，不自动删除旧 key
4. `module-dir` 下仅在当前模块内复用旧 key，不跨模块复用
5. 新 key 编号按该模块已有最大合法 `auto_xxx` 序号继续递增
6. 非标准 key 不影响新序号续增

### 5. report 能力

当前 `scan / extract / replace / run / apply` 都支持报告输出。

- `scan / extract / replace`：结构化 JSON report
- `run / apply`：支持执行总报告 `--report`

report 当前包含：

- `summary`
- `details`
- `samples`
- `review_priority`

### 6. Git 安全检查

`apply` 支持：

- `--git-check warn`（默认）
- `--git-check strict`
- `--git-check off`

当前只做执行前控制，不做：

- rollback
- snapshot
- patch 导出
- auto stash

## 四、当前扫描/替换边界

### 1. 文件类型

支持：

- `.js`
- `.ts`
- `.jsx`
- `.tsx`
- `.vue`

### 2. 注释过滤

当前已经实现注释区间过滤。

以下注释内容不会进入候选：

- `// ...`
- `/* ... */`
- `<!-- ... -->`

即：

- 不进入 scan candidates
- 不参与 extract
- 不参与 replace
- 不进入主 report 明细

### 3. 上下文分类层

当前已支持轻量上下文分类：

- `js_string`
- `template_attr_static`
- `template_text_static`
- `template_expr`
- `template_string`
- `unsafe_skip`

每个 candidate / report item 应至少包含：

- `context_type`
- `extractable`
- `replaceable`
- `skip_reason`（如有）

## 五、当前真正支持的最小国际化场景

这是当前已经收口后的真实边界，后续会话必须以此为准。

### A. template 中仅支持以下明确静态场景

1. `label="中文"`
2. `placeholder="中文"`
3. `<el-button>中文</el-button>`
4. `<el-table-column label="中文" />`
5. `{{ "中文" }}` 简单模板插值

替换示例：

- `label="用户名"` -> `:label="$t('...')"`
- `placeholder="请输入用户名"` -> `:placeholder="$t('...')"`
- `<el-button>保存</el-button>` -> `<el-button>{{ $t("...") }}</el-button>`
- `{{ "系统管理" }}` -> `{{ $t("...") }}`

### B. script 中仅支持以下白名单场景

1. `rules` 校验文案
   - `message: "中文"`
2. 明确标题位
   - `this.title = "中文"`
3. 成功消息提示
   - `this.$modal.msgSuccess("中文")`

---

## 六、当前明确不支持的内容

以下内容允许进入 scan/report，但默认不替换：

### template 不支持

- 任意动态表达式
- `template_expr` 扩展场景以外的复杂模板逻辑
- 三元表达式内部文本替换
- 动态绑定属性值
- 白名单外属性
- 白名单外标签静态文本
- 混合子节点文本

### script 不支持

- 任意 JS 字符串扫描替换
- 普通对象里的 `message: "中文"`
- `title: "中文"` 对象属性
- `dialogTitle = "中文"`
- 泛 `*Title` 变量赋值
- `this.$modal.msgError("中文")`
- `this.$modal.confirm("中文")` 的自动替换
- `this.$message.success/error("中文")`
- 模板字符串
- 字符串拼接
- 三元表达式内部替换

特别说明：

- `confirm("确认删除" + id + "吗")`
  - 允许 scan/report 标记
  - replace 必须跳过

## 七、每类已支持场景的规则说明

### 1. template: `label="中文"`

- 命中规则：
  - 属性名为 `label`
  - 属性值为明确静态中文字符串
- extract：
  - 允许提取
- replace：
  - 允许替换
- skip 条件：
  - `:label=`
  - 插值 / 动态表达式
  - 非静态值
- 风险说明：
  - 风险较低，但仅限静态字符串

### 2. template: `placeholder="中文"`

- 命中规则：
  - 属性名为 `placeholder`
  - 属性值为明确静态中文字符串
- extract：
  - 允许提取
- replace：
  - 允许替换
- skip 条件：
  - 动态绑定
  - 表达式
  - 非静态值
- 风险说明：
  - 风险较低

### 3. template: `<el-button>中文</el-button>`

- 命中规则：
  - 标签名为 `el-button`
  - 内部是单段纯静态中文文本
- extract：
  - 允许提取
- replace：
  - 允许替换为 `{{ $t(...) }}`
- skip 条件：
  - 含子节点
  - 含表达式
  - 含复杂文本拼接
- 风险说明：
  - 中低风险，必须保持标签白名单和纯文本限制

### 4. template: `<el-table-column label="中文" />`

- 命中规则：
  - 命中 `label="中文"`
- extract：
  - 允许提取
- replace：
  - 允许替换
- skip 条件：
  - 动态 `:label`
  - 非静态值
- 风险说明：
  - 风险低

### 5. script: `rules.message`

- 命中规则：
  - `message: "中文"`
  - 附近存在 `required / trigger / validator / pattern / min / max / type / rules` 等校验上下文
- extract：
  - 允许提取
- replace：
  - 替换为 `message: t("...")`
- skip 条件：
  - 普通对象里的 `message`
  - 语义不明的 `message`
- 风险说明：
  - 中风险，必须依赖上下文白名单

### 6. script: `this.title = "中文"`

- 命中规则：
  - 左侧明确为 `this.title =`
- extract：
  - 允许提取
- replace：
  - 替换为 `this.title = t("...")`
- skip 条件：
  - `const title = "中文"`
  - `dialogTitle = "中文"`
  - `title: "中文"`
- 风险说明：
  - 中风险，只允许最明确的标题位

### 7. script: `this.$modal.msgSuccess("中文")`

- 命中规则：
  - 固定调用名
  - 单个静态中文字符串参数
- extract：
  - 允许提取
- replace：
  - 替换为 `this.$modal.msgSuccess(t("..."))`
- skip 条件：
  - 拼接
  - 模板字符串
  - 多参数复杂形式
- 风险说明：
  - 风险较低

## 八、当前 CLI 使用方式

### 推荐日常试跑

```bash
i18n run --dir /path/to/module
```

说明：

- 执行 `scan`
- 执行 `extract`（仅内存，不写资源）
- 执行 `replace --dry-run`
- 不修改资源文件

### 推荐正式写入

```bash
i18n apply --dir /path/to/module --git-check strict
```

说明：

- 先提取并写资源
- 再执行代码替换
- 建议要求 Git 工作区干净

### 查看资源输出

```bash
i18n extract --dir /path/to/module
```

### 预演替换

```bash
i18n replace --dir /path/to/module --dry-run
```

### 推荐真实项目试跑方式

优先先跑小模块，不要一上来扫整个仓库根目录。

推荐顺序：

1. 新建分支
2. `i18n run --dir <模块目录> --report <文件>`
3. review report
4. 确认后再 `i18n apply --dir <模块目录> --git-check strict`

## 十、当前阶段明确不该做的事

## 十一、关于 key 的后续轻量方案

当前存在 `module.auto_xxx` 前缀偏粗的问题，但本阶段不要迁移历史 key，也不要推翻现有资源结构。

只允许做这种轻量思路：

- 后续新增 key 可以考虑更细的 `modulePrefix`
- 但不能影响历史 key
- 不能影响当前 replace 稳定性
